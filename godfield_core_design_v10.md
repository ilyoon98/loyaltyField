> 본 문서는 **캐릭터에 의존하지 않는 코어 시스템**만 다룬다. 캐릭터/전용장비/스킬은 별도 문서(2부)에서 이 코어 위에 얹는다.

## 1. 게임 개요

| 항목 | 내용 |
|---|---|
| 장르 | 다인 개인전(Free-for-all) 턴제 아이템 난투 |
| 인원 | 2 ~ 8명 (개인전, 팀 없음) |
| 플랫폼 | 웹 (HTML/CSS/JS), GitHub Pages 배포 |
| 네트워크 | Firebase Realtime DB 또는 Firestore (턴제라 실시간 압박 적음) |
| 승리 조건 | 마지막 1인 생존 (last man standing) |
| 1판 목표 시간 | 5~10분 |

### 핵심 재미 (이걸 검증한다)
- "누구를 때릴까"라는 **타겟 선택** 자체가 핵심 의사결정.
- 강한 사람이 두들겨 맞고(견제), 약자끼리 눈치 보고(연합), 막판에 뒤통수치는(배신·어부지리) 난투의 정치학.
- 캐릭터 전용 장비·스킬이 "누가 누구를 노리는가"에 변수를 더한다.

## 2. 기본 자원

| 자원 | 설명 | 기본값(조정 가능) |
|---|---|---|
| HP | 0이 되면 탈락 | **시작 50**, 최대 99 |

## 3. 아이템 분류 (코어 8~10종)

아이템은 크게 4종류. 캐릭터 전용 장비는 이 분류 체계 위에 특수효과를 얹는다.

### 3-1. 공격 (Weapon)
타겟 1명을 지명해 데미지. 타겟은 방어/반사 기회를 가진다.

| 예시 | 데미지 | 비고 |
|---|---|---|
| 단검 | 10 | 흔함 |
| 검 | 20 | 보통 |
| 도끼 | 35 | 무겁고 강함, 희귀 |
| 활 | 15 | (확장) 방어 무시 등 특수효과 여지 |

### 3-2. 방어 (Defense)
공격당했을 때 사용. 데미지를 막거나 줄인다.

| 예시 | 효과 |
|---|---|
| 방패 | 데미지 전부 무효 (1회) |
| 갑옷 | 데미지 절반 |

### 3-3. 반사 (Reflect)
공격당했을 때 사용. **데미지를 공격자에게 그대로 되돌린다.** ← 난투의 핵심 변수.

| 예시 | 효과 |
|---|---|
| 거울 | 받은 공격을 공격자에게 반사 |

> **설계 메모**: 반사가 있으면 "내가 A를 쳤는데 A가 반사 → 내가 맞음"이 발생. **v8부터 연쇄 허용** — 반사된 공격을 공격자가 다시 막거나 반사할 수 있다. 안전 상한 20회(13-4).

### 3-4. 전체공격 / 광역 (AoE)
타겟 지명 없이 **자신을 제외한 전원**에게 발동.

| 예시 | 효과 |
|---|---|
| 폭탄 | 전원에게 15 데미지 (각자 방어/반사/바운스 가능) |
| 독안개 | 전원 다음 턴 시작 시 5 데미지 (방어 불가) |

> **설계 메모**: 전체공격은 "선두 주자 깎기"와 "동시 다발 견제"를 만드는 양념. **v8부터 전체공격에도 반사·바운스 가능** (v7의 "ALL 반사 불가" 폐지). 단 되받아치기는 항상 단일 대상.

### 3-5. (확장) 보조/회복
| 예시 | 효과 |
|---|---|
| 물약 | HP 20 회복 |
| 행운 | 다음 턴 카드 2장 더 뽑기 |

## 4. 턴 흐름 (가장 중요한 규칙)

### 4-1. 턴 순서
- 입장 순서(또는 랜덤)로 고정된 순번. 1→2→3…→N→1 반복.
- 탈락한 플레이어는 순번에서 **스킵**.
- 생존자 1명이 되면 게임 종료.

### 4-2. 한 플레이어의 턴 (액티브 플레이어 = AP)
```
1. 카드 보충: 손패가 최대치 미만이면 1장 뽑는다.
2. 행동 선택 (택1):
   a) 공격 아이템 사용 → 타겟 1명 지명
   b) 전체공격 아이템 사용 → 타겟 자동(전원)
   c) 보조/회복 아이템 사용 (자신 포함 대상 선택)
   d) 패스 (아무것도 안 함, 카드 1장 획득)
3. 행동 1회로 턴 종료. (※ 빠른 진행을 위해 턴당 1행동 권장)
4. 다음 순번으로 넘어감.
```

> **설계 메모(대기시간 문제 대응)**: 8명 순차 턴제의 약점은 "내 차례까지 기다리는 시간". 이를 완화하는 두 장치:
> 1. **턴당 1행동**으로 각 턴을 짧게.
> 2. **남의 턴에도 내가 타겟이 되면 즉시 방어/반사 선택**을 하므로, 구경만 하는 게 아니라 계속 개입하게 됨. (아래 4-3)

### 4-3. 피격 처리 (남의 턴에 내가 타겟이 됐을 때)
공격 대상으로 지명된 플레이어(타겟 = TP)는 즉시 선택:
```
1. AP가 TP를 공격 (데미지 D)
2. TP의 응답 선택 (택1, 제한시간 30초):
   a) 방어 아이템 사용 (여러 장 누적 가능) → D 무효 또는 감소
   b) 반사 아이템 사용 → D를 AP에게 적용 (AP도 다시 응답 가능 — v8 연쇄)
   c) 튕기기(Bounce) → D를 랜덤 대상에게 전달 (연쇄 가능)
   d) 그냥 맞기 → HP -= D
3. HP 0 이하 → 탈락 처리
```

### 4-4. 전체공격 처리
```
1. AP가 전체공격 발동 (데미지 D)
2. AP 제외 전원이 각자 (a)방어 / (b)반사 / (c)튕기기 / (d)맞기 선택
   (반사·튕기기 결과는 단일 대상)
3. 각자 HP 정산, 0 이하 탈락
```

## 4-5. 캐릭터 선택 & 캐릭터 시스템

### 4-5-1. 설계 원칙
- **기본 스탯(HP, 손패, 카드 보충 등)은 전원 동일.** 캐릭터로 인한 스탯 차이는 없다.
- 캐릭터의 차별점은 오직 **"특정 아이템 카드를 사용할 때 발동하는 추가 효과(패시브)"** 하나뿐.
- 즉 **1 캐릭터 = 1 트리거 카드 + 1 추가 효과**. (프로토타입 단순화)
- 같은 캐릭터 **중복 선택 허용** (방 채우기 쉬움, 난투에 적합).

> **설계 메모(밸런스 핵심)**: 캐릭터 강함 = `트리거 카드의 출현 빈도 × 추가 효과의 크기`.
> 흔한 카드(단검)에 붙은 보너스는 자주 발동돼 강하고, 희귀 카드(도끼)에 붙으면 거의 발동 못 해 약하다.
> 따라서 "흔한 카드 = 작은 보너스 / 희귀 카드 = 큰 보너스"로 균형을 맞춘다.

### 4-5-2. 로비 / 캐릭터 선택 흐름
```
1. 방 입장 (roomId)
2. 캐릭터 선택 화면: 캐릭터 카드 목록에서 1명 선택 (중복 허용)
   - 각 캐릭터: 이름 / 컨셉 / 트리거 카드 / 추가 효과 설명 표시
3. 전원 캐릭터 선택 완료 시 방장이 게임 시작 버튼 활성화 → phase: playing
   (별도 "준비 완료" 토글 없음 — 선택 상태가 곧 준비 상태)
```

### 4-5-3. 캐릭터 데이터 구조
```json
"characters": {
  "char_chulsoo": {
    "name": "철수",
    "concept": "한 줄 컨셉",
    "trigger": { "item": "sword", "on": "attack" },  // 어떤 카드를, 언제 쓸 때
    "effect": { "type": "bonusDamage", "value": 5 }, // 무슨 추가 효과
    "desc": "검으로 공격 시 추가 데미지 +5"
  }
}
```

### 4-5-4. 추가 효과 타입 (effect.type) 예시 — 데이터로 관리
| type | 의미 | 발동 시점(on) |
|---|---|---|
| `bonusDamage` | 공격 데미지 +value | attack |
| `bonusDefense` | 방어 시 추가 경감 +value | defend |
| `reflectBonus` | 반사 데미지 +value | reflect |
| `drawOnUse` | 해당 카드 사용 시 카드 value장 추가 드로우 | use |
| `healOnUse` | 해당 카드 사용 시 HP +value 회복 | use |
| `pierce` | 해당 공격은 방어 무시 (반사만 가능) | attack |

> **설계 메모**: 추가 효과를 이렇게 `type + value` 데이터로 표준화하면, 새 캐릭터는 코드 수정 없이 데이터만 추가하면 된다. 으리챗 친구 수만큼 캐릭터를 찍어내기 쉬움.

### 4-5-5. 효과 적용 위치 (4장 턴 흐름에 끼워넣기)
- `on: "attack"` → 4-3 피격 처리에서 데미지 D 계산 시 보너스 가산.
- `on: "defend"` / `on: "reflect"` → 4-3 TP 응답 처리 시 적용.
- `on: "use"` → 행동 직후 드로우/회복 정산.

## 5. 게임 상태 데이터 구조 (온라인 동기화의 핵심)

> 온라인 멀티의 90%는 "이 객체를 어떻게 주고받느냐"의 문제다. 이 구조를 Firebase에 통째로 올리고 동기화한다.

```json
{
  "roomId": "ABCD",
  "phase": "playing",          // waiting | playing | finished
  "turnOrder": ["p1","p2","p3","p4"],
  "currentTurnIndex": 0,        // turnOrder상 현재 AP 위치
  "action": null,               // 진행 중 액션 큐 (아래 14-4)
  "players": {
    "p1": {
      "name": "철수",
      "characterId": "char_xxx",
      "hp": 50,
      "alive": true,
      "hand": ["kalbang","painkill","mirror","cook","protein"]
    }
  },
  "log": [ "p1 → p3 공격(칼빵 5)", "p3 무지개 반사 → p1 5 피해" ],
  "winner": null,
  "voided": false
}
```

> **설계 메모**: 모든 클라이언트는 이 객체 하나만 바라본다. "내가 AP인가? → 행동 UI 표시", "action.await.target이 나인가? → 방어/반사 UI 표시" 식으로 화면을 그린다. **누가 무엇을 할 수 있는지는 전부 이 상태에서 파생**되므로 동기화가 단순해진다.

## 6. 온라인 진행/예외 처리 체크리스트

> **v10 주의**: 아래는 설계 당시 체크리스트다. **실제 구현 현황은 14장(as-built)** 을 보라 — 응답 타임아웃(14-7), 새로고침 복구(14-8), 끊김 처리 5c-2~4(14-10), 방장 이양(14-10), 유령 방 정리(14-12)가 모두 코드로 구현됨.

| 상황 | 처리 |
|---|---|
| 응답 제한시간 초과 | 자동 "맞기" 처리 후 진행 (게임이 멈추지 않게) |
| 플레이어 접속 끊김 | 5c-2: 턴 스킵 / 5c-4: 30초 후 탈락 |
| 새로고침 | roomId 기억 → 상태 재로딩 (state가 DB에 있으니 복구 가능) |
| 동시 입력 충돌 | AP만 행동 가능하도록 트랜잭션 재검증 |
| 방장 나감 | 5c-4 탈락 처리 시 연결된 생존자에게 권한 이양 |
| 게임 종료 후 방 | leaveRoomAndReload로 즉시 삭제, sweepGhostRooms로 주기 정리 |

## 7. 밸런스 초기값 (테스트하며 조정)

| 변수 | 초기값 | 조정 방향 메모 |
|---|---|---|
| 시작 HP | **50** | 게임이 너무 길면 ↓ |
| 최대 HP | **99** | 회복 상한 |
| 손패 최대 | 6 | 운 요소 ↑ 원하면 ↓ |
| 강무기 출현율 | 낮게 | 도끼 자주 나오면 난투가 너무 빨리 끝남 |
| 방어/반사 비율 | 공격:방어:반사 ≈ 5:3:2 | 반사 너무 많으면 아무도 공격 안 함 |
| 전체공격 비율 | 전체의 10% 이하 | 너무 많으면 지명 공격의 정치성이 죽음 |

> **설계 메모**: "반사가 너무 흔하면 아무도 공격을 안 하는 교착"이 다인전 난투의 대표적 붕괴 패턴. 반사는 희귀하게.

## 8. 개발 단계 로드맵 (망하지 않는 순서)

1. **로컬 N인 완성** — 한 화면에서 더미 플레이어들 상대로 턴 루프·아이템·반사·전체공격까지 전부 작동 (네트워크 0%)
2. **상태 객체 확정** — 위 5장 JSON으로 모든 게임 진행을 표현 가능한지 검증
3. **Firebase 연결** — 방 생성/입장, 상태 객체 동기화
4. **턴·피격 동기화** — AP 행동, 큐 네트워크화, 반응 입력 흐름
5. **예외 처리** — 6장 체크리스트
6. **(이후) 캐릭터/전용장비/스킬 얹기** — 2부 문서

> 1번이 끝나면 이미 "재밌는지"는 검증된다. 온라인은 그 다음.

## 9. 다음 작업 (2부에서 채울 것)

캐릭터 **시스템**(선택 흐름·데이터 구조·효과 타입)은 4-5장에서 코어로 확정됨.
2부는 그 위에 얹을 **실제 캐릭터 콘텐츠**다.

- 으리챗 캐릭터별: 이름 / 컨셉 / 트리거 카드(`trigger.item`) / 추가 효과(`effect.type`+`value`)
- 캐릭터 밸런스 점검: 트리거 카드 출현 빈도 × 보너스 크기 (4-5-1 메모 기준)
- (선택) 트리거 카드가 희귀 카드일 경우 발동 기회를 어떻게 보장할지

→ 으리챗 친구들 특징을 알려주면 이 코어 위에 얹어 캐릭터 콘텐츠를 작성한다.

## 10. 으리챗 캐릭터 콘텐츠 (2부)

> 전원 기본 스탯 동일. 차별점은 트리거 카드 1종 + 추가 효과 1개.
> `effect.type`은 4-5-4 표준 타입 사용. 밸런스는 4-5-1 원칙(흔한 카드=작은 보너스 / 희귀 카드=큰 보너스).

| 캐릭터 | 컨셉(특징 반영) | 트리거 카드 | 발동 | 효과 | 비고 |
|---|---|---|---|---|---|
| **최초** | 유명한 강아지 '뭉치'의 주인 — 든든한 동료 | 단검(흔함) | attack | `bonusDamage +4` | 흔한 카드라 자주 발동, 보너스는 소폭 |
| **기종** | 허리 부상 — 버티는 탱커형 | 갑옷(보통) | defend | `bonusDefense +10` | 맞아도 잘 버팀 |
| **규형** | 토목·콘크리트 석사, 역마살 | 방패(보통) | defend | `reflectBonus +8` | 단단한 콘크리트=방어/되받음 |
| **지원** | 모든 게임을 잘함 — 만능형 | 패스/드로우(행운) | use | `drawOnUse +1` | 손패 회전이 빨라 선택지 많음(올라운더) |
| **소민** | 건설 설계→경비원, 지키는 사람 | 방패(보통) | defend | `drawOnUse +1`(막은 뒤 1장) | 방어하며 자원 확보 |
| **준형** | 인싸 '준형전자', 견적의 달인 | 활(보통) | attack | `pierce`(방어 무시) | 정확히 약점을 찌름 |
| **종문** | 으리챗 막내 — 막내 보정(운) | 물약(보통) | use | `healOnUse +10`(추가 회복) | 어리고 회복 빠름 |
| **종원** | 요리사 — 회복 특화 | 물약(보통) | use | `healOnUse +15` | 회복량 최고, 단 공격력 보너스 없음 |
| **현일** | 운동광 — 힘 세고 공격적 | 검(보통) | attack | `bonusDamage +8` | 보통 빈도 카드에 중간 보너스 |
| **기훈** | 수많은 게임 섭렵 — 노련한 변수 | 도끼(희귀) | attack | `bonusDamage +15` | 희귀 카드라 발동 드묾, 터지면 강력 |

### 10-1. 밸런스 점검 메모
- **자주 발동 / 약한 보너스**: 최초(단검), 지원(드로우) → 꾸준함.
- **가끔 발동 / 강한 보너스**: 기훈(도끼 +15) → 한 방 로망, 발동 기회가 적어 OP 아님.
- **방어/생존형**: 기종·규형·소민 → 난투에서 오래 살아남아 어부지리 노림.
- **회복형**: 종문·종원 → 장기전 강함, 단 직접 공격 보너스 없음(견제 대상 덜 됨).
- **주의**: 규형의 효과는 처음에 너무 복잡하게 잡혀 단순화함. 반사는 희귀 자원이므로 `reflectBonus`는 1명(규형)으로 제한해 교착 방지.

### 10-2. 캐릭터 데이터 (JSON 예시 — 그대로 코드에 사용 가능)
```json

  "choi":    { "name":"최초", "trigger":{"item":"dagger","on":"attack"}, "effect":{"type":"bonusDamage","value":4} },

  "junhyung":{ "name":"준형", "trigger":{"item":"bow","on":"attack"},    "effect":{"type":"pierce","value":1} },
  "jongmun": { "name":"종문", "trigger":{"item":"potion","on":"use"},    "effect":{"type":"healOnUse","value":10} },
```

## 11. 통합 카드 데이터 스펙 (최종)

> 카드와 캐릭터 보너스를 한 테이블에서 관리한다. 한 행 = 카드 1종.
> 캐릭터 효과는 `Trigger`(어느 캐릭터) + `Plus Type`/`Plus Value`(무슨 보너스)로 같은 행에 얹는다.

### 11-1. 컬럼 정의

| 컬럼 | 값 | 의미 |
|---|---|---|
| index | 정수 | 카드 고유 번호 |
| Name | 문자열 | 카드 이름 |
| Type | ATK / DEF / HEAL / ALL / REFLECT / Bounce | 카드가 하는 일 |
| Value | 정수 | 기본 수치 (데미지/방어/회복량). Bounce는 0 (자체 데미지 없음) |
| Probability | 0~1 | **명중 확률**. 1이면 무조건 적중 |
| **Timing** | **MyTurn / OnHit** | **사용 가능 시점** (DEF·REFLECT·Bounce=OnHit, 그 외=MyTurn) |
| Trigger | 캐릭터명 / 없음 | 이 카드에 보너스를 받는 캐릭터 |
| Plus Type | ProbabilityUP / ValueUP / HEAL / Target | Trigger 캐릭터가 쓸 때 추가되는 효과 |
| Plus Value | 정수/소수 | 추가 효과의 크기 |

### 11-2. Type 정의

| Type | 동작 | Timing | 비고 |
|---|---|---|---|
| ATK | 지명 1명 공격 | MyTurn | |
| ALL | 자신 제외 전원 공격 (대상별 개별 명중판정) | MyTurn | |
| HEAL | **자신 포함 대상 1명** HP 회복 (타겟 선택) | MyTurn | |
| DEF | 들어온 데미지 무효/경감 (여러 장 누적 가능) | OnHit | |
| REFLECT | 들어온 데미지를 공격자에게 그대로 반사 (연쇄 허용) | OnHit | |
| Bounce | 들어온 데미지를 랜덤 대상(생존자, 본인 포함)에게 떠넘김 (연쇄 허용) | OnHit | 본인에게 돌아오면 즉시 맞기 |

### 11-3. Plus Type 정의

| Plus Type | 효과 | 적용 시점 |
|---|---|---|
| ProbabilityUP | 명중률 + Plus Value | Trigger 캐릭터가 그 카드로 공격 시 |
| ValueUP | 기본 수치 + Plus Value | 사용 시 |
| HEAL | 카드 효과에 더해 HP + Plus Value 회복 | 사용 시 |
| Target | 동시 대상 수 + Plus Value (명) | 사용 시 (Bounce 등) |

### 11-4. 명중/반응 처리 규칙 (v8 갱신)

- **명중 판정**: 공격 시 `Probability`로 적중 여부 굴림. 빗나가면 데미지 0, **카드는 소모됨**.
- **ALL 공격**: 대상마다 개별로 명중 판정 (뭉치 짖기 0.5 → 8명에게 각각 50% 굴림).
- **피격 응답 (한 번의 피격에 택1 종류)**: 타겟은 아래 중 **한 종류**를 선택해 확정한다.
  - **방어(DEF) 누적**: 방어 카드를 **여러 장** 사용 가능. 합산 방어량 ≥ 데미지면 0. (예: 20 데미지에 10방어 2장 → 0)
  - **반사(REFLECT)**: 받은 데미지를 공격자에게 되돌림.
  - **튕기기(Bounce)**: 받은 데미지를 랜덤 대상(생존자, 본인 포함)에게 전달.
  - **그냥 맞기**.
  - ※ 방어와 반사/튕기기를 **섞어 쓰지 않는다**(한 응답엔 한 종류만).
- **연쇄 허용 (v8 핵심 변경)**: 되받아친 데미지(반사·튕기기 결과)에 대해 **받은 쪽이 다시 반사/튕기기/방어로 응답 가능**. 즉 A→B 공격을 B가 반사 → A가 또 반사 → … 연쇄가 이어진다.
  - **전체공격(ALL)도 반사·튕기기 가능** (v7의 "ALL 반사 불가" 폐지). 단 되받아친 데미지는 항상 **단일 대상**.
  - **튕기기가 본인에게 돌아오면**: 다시 튕기지 못하고 **즉시 맞는다**(자기 자신에 대한 추가 반응 불가).
  - **종료 조건**: ①응답할 OnHit 카드가 손패에 없음 ②그냥 맞기 선택 ③대상 사망 ④**안전 상한 20회 도달 시 21번째는 강제 맞기**.
- **카드 보충 타이밍 (v8 변경)**: 카드는 **카드 사용 즉시가 아니라, 한 행동(공격~모든 연쇄 종료)이 완전히 끝난 뒤** 보충한다. → 연쇄 중에는 손패가 고정되어 유한 자원이 자연 종료를 보장한다.

### 11-5. 현재 카드 목록 (업로드 테이블 기준)

| idx | Name | Type | Value | Prob | Timing | Trigger | Plus Type | Plus Value |
|---|---|---|---|---|---|---|---|---|
| 1 | 뭉치 짖기 | ALL | 10 | 0.5 | MyTurn | 최초 | ProbabilityUP | 0.25 |
| 2 | 칼빵 | ATK | 5 | 1 | MyTurn | — | — | — |
| 3 | 요리하기 | HEAL | 5 | 1 | MyTurn | 종원 | ValueUP | 2 |
| 4 | 진통제 | DEF | 10 | 1 | OnHit | 기종 | ValueUP | 5 |
| 5 | 프로틴 | ATK | 5 | 1 | MyTurn | 현일 | HEAL | 2 |
| 6 | 무지개 반사 | REFLECT | 0 | 1 | OnHit | — | — | — |
| 7 | 유수암쇄권 | Bounce | 0 | 1 | OnHit | 기훈 | Target | 2 |

> **남은 캐릭터 미배치**: 규형·지원·소민·준형·종문 — 카드가 더 추가되면 Trigger로 배치.
> **밸런스 메모**: 프로틴(ATK+HEAL on 현일)은 "때리면서 회복"이라 강력 → Plus Value 작게 유지. Bounce는 Value 0이므로 단독 가치 없음, 반드시 피격 반응으로만 의미.

## 12. 연출/진행 시스템 (갓필드식 단계별 재생)

> 문제의식: 로그 텍스트만으로는 "누가 누구에게 뭘 썼고 어떻게 막혔는지"가 안 보인다.
> 갓필드처럼 **한 액션을 여러 단계로 쪼개, 화면 중앙에서 한 박자씩 재생**한다.
> 진행 방식은 **순차 턴제 유지(A안)**. 동시 선택제 아님.

### 12-1. 핵심 구조: 이벤트 큐 + 재생기

로직(누가 누구에게 얼마)과 연출(어떻게 보여주나)을 분리한다.

1. **이벤트 생성**: 한 액션이 일으킬 일들을 순서대로 배열에 쌓는다.
2. **재생기(player)**: 큐를 하나씩 꺼내 화면에 보여주고, 사람 입력이 필요하면 멈추고, 자동이면 일정 시간(0.8~1.2s) 뒤 다음으로.

```javascript
// 예: 상대C가 나에게 칼빵 → 내가 무지개 반사
const queue = [
  { kind:"announce", actor:"ai3", card:"kalbang", target:"you" }, // 중앙에 "상대C → 나: 칼빵!"
  { kind:"hitcheck", target:"you", prob:1 },                       // 명중 판정 (빗나가면 MISS 후 종료)
  { kind:"askDefense", target:"you", incoming:5 },                 // ★사람 입력 대기 (방어/반사/바운스/맞기)
  { kind:"react", actor:"you", card:"mirror" },                    // "반사!" 중앙 연출
  { kind:"damage", target:"ai3", amount:5, label:"반사" },          // 데미지 숫자 튀어오름
];
```

> **온라인 확장성**: 이 구조는 그대로 온라인으로 간다. owner가 queue를 계산해 DB에 올리면 모든 클라가 똑같이 재생. "askDefense"만 해당 플레이어 클라에서 입력받아 DB로 전송.

### 12-2. 이벤트 종류 (kind)

| kind | 의미 | 멈춤? |
|---|---|---|
| announce | 공격자·카드·대상을 중앙에 크게 표시 | 자동(짧게) |
| hitcheck | 명중 굴림. 실패 시 MISS 연출 후 이 대상 종료 | 자동 |
| askDefense | 피격자에게 방어/반사/바운스/맞기 선택 요청 | **입력 대기** |
| react | 피격자의 대응을 중앙에 표시("막음!"/"반사!"/"떠넘김!") | 자동 |
| damage | 데미지 숫자가 HP바 위로 튀어오름 | 자동 |
| selfheal | 시전자 스킬 회복 연출 | 자동 |
| counterAnnounce | 반사/바운스 공수전환 연출 | 자동 |
| clearCR / clearAll | 구역 초기화 | 자동 |
| restoreAllAnnounce | 전체공격 되받아치기 후 원 시퀀스 재개 | 자동 |

### 12-3. 전체공격(ALL) 단계별 처리 ★중요

ALL 카드는 대상 수만큼 위 흐름이 반복된다. 한꺼번에 처리하지 않는다.

```
뭉치 짖기(ALL) 사용 + "완료"
  → announce: "뭉치 짖기!"
  → [대상A] hitcheck → 명중 → askDefense(방어/반사/바운스/맞기) → react → damage
  → [대상B] hitcheck → 빗나감 → MISS 표시, 다음으로
  → [대상C] hitcheck → 명중 → askDefense → react → damage
  → 종료, 다음 턴
```

> **ALL도 반사·바운스 가능 (v8)**: 대상이 반사를 쓰면 그 자리에 되받아치기 패턴이 splice 삽입되고, 원래 전체공격은 취소되지 않고 남은 대상에게 계속 진행.

### 12-5. 화면 레이아웃 — 3구역 액션 디스플레이 ★v7 갱신

한 번의 공격 처리(=한 패턴)를 **3구역**으로 나눠 화면 중앙에 카드로 보여준다.

```
 ┌──────────┬──────────┬──────────┐
 │   왼쪽    │   중앙    │  오른쪽   │
 │  시전자   │  피격자   │   결과    │
 │ 행동 카드 │ 대응 카드 │  데미지   │
 └──────────┴──────────┴──────────┘
```

| 구역 | 내용 | 예시 |
|---|---|---|
| **왼쪽** | 시전자의 행동 카드 + "시전자 → 대상" 표시 | A의 칼빵 (A → B) |
| **중앙** | 피격자의 대응 카드 | B의 진통제(방어) / 무지개 반사 / 유수암쇄권 |
| **오른쪽** | 처리 결과 | 데미지 0 / 데미지 5 / MISS |

**패턴 진행**
1. 왼쪽에 시전 카드 + 대상 표시.
2. 중앙에 피격자 대응 카드 (방어/반사/바운스). 그냥 맞으면 중앙 비움 또는 "맞음".
3. 오른쪽에 최종 데미지 숫자.
4. **한 패턴 종료 → 3구역 카드 모두 사라짐 → 다음 패턴.**

> 기존 12-2의 announce/react/damage 이벤트가 각각 왼쪽/중앙/오른쪽 구역 갱신에 대응한다.
> 카드 = 타일(아이콘+이름+수치). 로그는 하단에 보조로 축소.

### 12-6. "완료 누르면 재생" 흐름

- 내 턴: 카드 선택 → (대상 지명 필요 시 대상 클릭) → **카드 클릭으로 즉시 큐 재생 시작**.
- 재생 중에는 손패·버튼 비활성화. askDefense 단계에서만 해당 플레이어 입력 활성화(손패 인라인 반응 바).
- 재생이 끝나면 다음 턴.

### 12-7. 반사·바운스 공수전환 연출 ★v7 신설

REFLECT(무지개 반사)·Bounce(유수암쇄권)는 피격자가 시전자로 **역전**된다.
이때 **3구역을 새로 한 번 더 사용**해 "되받아치는 공격"을 별도 패턴으로 재생한다(해석 1 = 큐에 패턴 삽입).

#### 12-7-1. 반사(REFLECT) — A가 B 공격 → B가 반사 (v8: 연쇄)
```
[패턴 1] 왼쪽: A의 공격카드(A→B) | 중앙: B의 무지개 반사 | 오른쪽: (반사 발생, B 데미지 0)
   ↓ 공수 전환 — 큐에 새 패턴 삽입
[패턴 2] 왼쪽: 반사된 데미지 카드(B→A) | 중앙: (A의 응답) | 오른쪽: A 데미지 N
```
- 반사로 되돌아간 데미지는 **단일 대상(A)에게만** 적용.
- **A는 다시 응답 가능** (v8: 반사를 또 반사/방어/맞기). 즉 패턴 2 끝에 다시 askDefense가 붙어 연쇄.
- **종료**: 응답 카드 없음 / 맞기 선택 / 사망 / 안전상한 20회.

#### 12-7-2. 튕기기(Bounce) — B가 떠넘김 (v8: 연쇄)
```
[패턴 1] 왼쪽: 원공격카드 | 중앙: B의 유수암쇄권 | 오른쪽: (떠넘김 발생)
   ↓ 대상 수만큼 새 패턴 삽입 (기훈 Target+2면 다중)
[패턴 2] 왼쪽: 떠넘긴 데미지(B→랜덤대상1) | 중앙: (대상1 응답) | 오른쪽: 데미지 N
```
- 떠넘김 대상은 **생존자 중 랜덤**(본인 포함). 각 대상은 **다시 응답 가능**(연쇄).
- **단, 튕긴 데미지가 본인(B)에게 돌아오면 즉시 맞는다** — 자기 자신엔 추가 반응 불가.
- **종료**: 위 반사와 동일.
- **baking**: 튕기기 대상은 큐 생성 시점에 확정(RNG 고정). 재생 시점 재롤링 금지 → 온라인 결정성 보장.

#### 12-7-3. ★전체공격 중 반사/바운스가 끼어들 때 (핵심 규칙)

전체공격(ALL) 시퀀스 도중 한 대상이 반사/바운스를 쓰면, **그 되받아치기 패턴을 그 자리에 끼워넣되, 원래 전체공격은 취소되지 않고 남은 대상에게 계속 진행**한다.

```
A가 뭉치 짖기(ALL) → 대상 [B, C, D] 순차 처리
  [A→B] 처리 중 B가 무지개 반사 사용
     → (끼어듦) [B→A] 단일 데미지 패턴 재생, A는 재응답 가능
  [A→C] 계속 진행 (예정대로)
  [A→D] 계속 진행
  → A의 턴 종료
```

> **구현 메모(이벤트 큐)**: ALL은 이미 큐에 `[A→B, A→C, A→D]`로 쌓여 있다.
> `A→B` 처리 중 반사가 발생하면 **그 직후 위치에 `[B→A 단일공격]` 패턴을 splice 삽입**한다.
> 삽입이 끝나면 큐에 남은 `A→C, A→D`가 그대로 이어진다.

## 13. 연쇄·방어중첩 상세 (v8 신설)

### 13-1. 피격 응답 모델 (한 종류만)
한 번의 피격에 대해 타겟은 아래 **한 가지**를 골라 "완료"로 확정한다(12-6 흐름과 동일).

| 응답 | 카드 수 | 결과 |
|---|---|---|
| 방어 | 1장 이상 누적 | 합산 방어량만큼 데미지 감소. ≥데미지면 0 |
| 반사 | 1장 | 데미지를 공격자에게 되돌림 (연쇄) |
| 튕기기 | 1장 | 데미지를 랜덤 생존자에게 전달 (연쇄), 본인이면 즉시 맞기 |
| 맞기 | 0장 | 데미지 그대로 수용 (연쇄 종료) |

> **방어와 반사/튕기기는 섞지 않는다.** (v8 결정: 한 응답엔 한 종류)

### 13-2. 방어 중첩 처리
- 응답 UI에서 방어 카드를 **여러 장 탭해 누적 선택** → "방어 완료".
- `최종데미지 = max(0, 들어온데미지 − Σ방어량)`. 사용한 방어 카드는 전부 소모.
- 예: 20 데미지 ← 진통제(10) 2장 → 0 데미지, 카드 2장 소모.

### 13-3. 연쇄(반사/튕기기) 처리 — 큐 재귀 삽입
- 반사/튕기기가 나오면 되받아치기 패턴 `[counterAnnounce → damage → askDefense]`를 큐에 splice 삽입.
- **끝에 askDefense가 다시 붙으므로** 받은 쪽이 또 응답 가능 → 자연 재귀.
- **카드 보충은 연쇄가 완전히 끝난 뒤** 양쪽 모두 1회만 정산(11-4). 연쇄 중 손패 고정 → 유한 자원이 종료 보장.

### 13-4. 안전 상한 (연쇄 20회)
- 한 "행동"에서 발생한 되받아치기 패턴 수를 카운트.
- **20회 도달 시 21번째 응답은 강제 "맞기"** 처리하고 종결. (손패가 남아 있어도)
- 화면: "연쇄 종료!" 짧게 표시 후 마지막 대상이 데미지 수용.
- 목적: 정상 플레이에선 손패(최대 6장)가 먼저 떨어져 도달 불가 → 버그성 무한루프 방지용 최후 안전판.

### 13-5. 전체공격 연출 보강
- ALL 처리 시 **중앙 구역에 명중/빗나감 표시**: 명중="명중!", 실패="빗나감 💨".
- 단 해당 대상이 **반사/튕기기로 응답하면, 중앙의 명중표시를 대응 카드가 덮어쓴다**(중앙은 대응 카드 우선).
- 즉 중앙 구역 우선순위: 대응 카드(반사/튕기기/방어) > 명중·빗나감 표시.

## 14. 온라인 구현 (as-built) — v9 신설, v10 일부 현행화

> 본 장은 **구상이 아니라 `index.html`에 실제 구현된 코드 기준**이다. 5·6장이 설계 의도라면, 14장은 그 의도가 어떻게 코드로 굳었는지를 기록한다. 코드가 갱신되면 이 장을 먼저 현행화한다.

### 14-1. 네트워크 모델 개요
- **백엔드**: Firebase Realtime Database. 방 하나 = `rooms/{code}` 단일 객체. 모든 클라이언트가 이 노드를 `on('value')`로 구독하고, 화면은 전부 이 상태에서 파생(누가 AP인지·피격 대상인지 등).
- **권위 모델**: **owner = 현재 AP(액티브 플레이어)**. 행동을 일으킨 사람이 큐를 만들고 랜덤을 baking 해 DB에 올리면, 전원이 동일하게 재생한다. 끊김 같은 무주공산 상황만 별도의 **복구 권위자(recovery authority)** 가 처리한다.
- **핵심 원칙**: 온라인 상태 변경(턴/HP/alive/phase/winner/host)은 **owner 또는 복구 권위자의 단일 원자 write**로만. 클라가 각자 자기 상태를 바꾸면 desync.

### 14-2. DB 스키마 `rooms/{roomId}`
| 필드 | 내용 |
|---|---|
| `host` | 방장 uid. 게임 중엔 런타임 권한 없음(대기실 "게임 시작" 버튼 권한 + 👑 표시용). |
| `phase` | `waiting` / `playing` / `finished` |
| `players/{uid}` | `{ name, characterId, hp, alive, hand[] }` |
| `turnOrder` | uid 배열(턴 순서) |
| `currentTurnIndex` | 현재 AP 인덱스. **모든 클라가 자기 턴 여부를 이 값으로만 판정** |
| `log` | 로그 배열 |
| `winner` | 우승자 uid(또는 null) |
| `voided` | 전투 동시 전멸 무효 플래그(5c-4) |
| `action` | 진행 중 액션 큐(아래 14-4) |
| `presence/{uid}` | `{ connected, lastSeen }` — onDisconnect로 서버가 갱신(아래 14-9) |
| `createdAt` | 방 생성 서버 타임스탬프. 유령 방 정리 age 기준 |

### 14-3. 식별 / 세션
- `myUid`: `sessionStorage.euri_uid`(없으면 생성). 같은 탭 새로고침 시 동일 uid 유지 → 복구의 키.
- **방 코드**: 4자리 영숫자, 생성 시 중복 회피. 입장은 `phase==='waiting'` & 4명 미만일 때만.

### 14-4. 액션 큐 동기화 (owner 권위) — 기획 12장의 네트워크판
- **진입**: owner가 카드 확정 → `startOnlineAction` → 큐 빌드(ATK/HEAL/ALL) → `claimAndUploadQueue`.
- **claim 트랜잭션**: "내가 진짜 현재 AP인가 + 진행 중 action 없는가"를 **원자적으로 재검증**한 뒤에만 `action = { seq, queue, version, await:null, input:null, done:false, ownerUid }`를 올린다. 아니면 abort. → **동시 행동·중복 업로드 경합 차단**.
- **재생**: 전원의 `onRoomData`가 `action.seq` 증가를 감지해 동일 큐 재생(owner 포함 같은 경로).
- **피격 입력**: `askDefense` 단계에서 owner가 `action/await = {index, target, attacker, item, incoming, aoe}` 게시 → 타겟 클라의 `watchAwait`(상설 리스너)가 손패 인라인 반응 바 표시 → `submitOnlineInput`이 `action/input`에 선택 기록 → owner가 읽어 큐에 반응 이벤트 splice 후 resolved 표시.

### 14-5. 정산 — `commitSettlement` 단일 write
재생이 끝나면 owner가 **한 번의 `update`** 로 확정한다: 손패 보충 + `currentTurnIndex`(다음 생존자) + `phase`/`winner`/`voided` + `action/done:true`. 클라는 이 값만 보고 다음 턴을 그린다. (패스는 `onlinePass` 트랜잭션.)

### 14-6. watchAwait — 재생 가드와 분리된 상설 리스너
피격 응답 UI는 `action/await`를 **별도 상설 구독**으로 처리한다(재생 루프 안이 아님). 덕분에 재생 중에도, 새로고침 직후에도 내 차례의 UI가 안정적으로 뜨고, await가 해제되면(resolve 또는 owner 이탈로 abort) 자동으로 닫힌다.

### 14-7. 견고화 5a — 응답 타임아웃
`RESPONSE_TIMEOUT = 30000ms`. 피격자가 응답을 안 하면 owner가 자동 `take`(맞기)로 resolve → 게임 멈춤 방지. 타겟 클라엔 코스메틱 카운트다운 표시.

### 14-8. 견고화 5b — 새로고침 복구
- `saveSession`이 `sessionStorage`에 `euri_room`/`euri_name` 저장. 페이지 로드 시 `tryRejoinOnline`이 자동 복귀.
- 복구 취소 조건: 방 없음 / 내가 멤버 아님 / `phase==='finished'`.
- **in-flight 처리**: 복귀 시 `onlineSeq`를 현재 seq로 맞춰 재생 중복을 막고, 끊긴 게 **나(=AP/owner)였다면** 사라진 내 재생 루프 때문에 묶인 모두를 풀기 위해 `action/done:true`로 취소(직전 상태로 깔끔히 복귀).

### 14-9. presence (5c-1) — 끊김 감지 인프라
- `.info/connected` 구독 + `onDisconnect().set({connected:false, lastSeen: ServerValue.TIMESTAMP})`로 **서버가 끊김 시각을 박는다**. 재접속 시 자동 재무장.
- `presence` 노드는 players/action/commit 경로와 **분리된 별도 상설 리스너**(`onlineReplaying` 가드와 무관 → 재생 중에도 감지). UI에 🔌·흐림 처리.

### 14-10. 끊김 처리 권위 모델 (5c-2 ~ 5c-4)
무주공산(AP가 끊겨 아무도 못 미는 상황)은 **단일 복구 권위자**가 처리한다.
- **`amRecoveryAuthority()`**: turnOrder상 **첫 번째 alive+connected** 한 명만 시도(부하·중복 최소화 — 최적화일 뿐).
- **진짜 보장 = 트랜잭션 재검증**: 실제 상태 변경은 `rooms/{roomId}` 트랜잭션 안에서 전제(여전히 끊김·유예 경과·alive 등)를 다시 확인하고 통과할 때만 commit. 중복·동시 시도는 모두 abort. **멱등**.
- **권위자 승계**: 권위자가 끊기면 다음 alive+connected가 다음 presence 이벤트에서 자동 인수.

| 단계 | 유예 | 동작 | 트리거/방어 |
|---|---|---|---|
| 5c-2 | `SKIP_GRACE` 5s | idle AP 끊김 → **턴 스킵**(alive 불변) | `checkAPRecovery`→`skipDisconnectedAP` 트랜잭션 |
| 5c-3 | `ABORT_GRACE` 0s | **행동 중** owner 끊김 → **action abort**(`done:true`, await 클리어 → 묶인 spectator 즉시 해제) | `checkOwnerAbort`→`abortOwnerAction`. 턴은 안 건드리고 5c-2에 위임 |
| 5c-4 | `ELIM_GRACE` 30s | 끊김 지속 → **탈락(alive=false)** + 승패 재계산 + host 이양 | `checkElimination`→`eliminateDisconnected` 트랜잭션 |

**5c-4 동시성 방어 핵심**:
- in-flight action(`action.done===false`) 중에는 탈락 **보류**(큐가 그 uid를 target으로 참조 중일 수 있어 desync 방지). 5c-3가 owner 끊김은 즉시 abort하므로 보류는 짧다.
- 다중 끊김은 한 트랜잭션에서 **일괄 탈락**. `alive`/`winner`/`phase`/`currentTurnIndex`/`host`를 **한 write**에서만 변경.
- 복귀 race: 유예 중 재접속하면 presence 변화로 타이머 취소 + 트랜잭션이 `connected`/`lastSeen` 최종 재검증해 커밋 직전 복귀도 abort.

### 14-11. 승패 엣지 케이스
- **끊김으로 `alive ≤ 1`**: 1명 남으면 그 사람 우승. 동시 전멸(0명)이면 **마지막에 끊긴 사람(`lastSeen` 최대)이 우승**.
- **전투로 `alive = 0`**(전체공격 등 HP 동시 0): `voided=true` → 승자 없음, **"그리고 아무도 없었다"** 무효 처리(`commitSettlement`/`checkWin`/`showWin`).
- 구분 기준: **끊김 경로 = 우승자 있음 / 전투 경로 = 무효**.

### 14-12. 게임 종료 후 방 정리 — 구현 완료 (v10 현행화)

**Stage 1 — `sweepGhostRooms` (유령 방 청소)**
- 새 방을 만들 때 fire-and-forget으로 호출. 모든 방을 스캔해 아래 조건 충족 시 삭제:
  - `presence`에 아무도 `connected:true` 없음
  - 마지막 이탈(`lastSeen` 최대, 없으면 `createdAt`) 이후 `GHOST_GRACE`(120s) 또는 `FIN_GRACE`(10s) 경과
- 삭제는 **트랜잭션 재검증**으로 멱등(이미 없으면 no-op, rejoin 중이면 abort).

**Stage 2 — `leaveRoomAndReload` (게임 후 나가기)**
- 승리 화면 "나가기" 버튼 → `phase==='finished'`인 방만 즉시 삭제 → 새로고침.
- 리스너 먼저 off 후 삭제(null 콜백 방지). 실패해도 Stage 1 sweep이 나중에 처리.

**`rematchOnline` (재대결)**
- 승리 화면 "다시 하기" 버튼(온라인) → `phase==='finished'` 방을 `waiting`으로 리셋.
- 플레이어 이름 유지, hp/alive/hand 제거, winner·voided 초기화. action seq는 유지(단조 증가).
- 성공 시 전원이 대기실(캐릭터 선택)로 복귀.
